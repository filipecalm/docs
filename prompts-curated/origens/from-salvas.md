### Front-end moderno — competências além do básico

**Fonte:** `Screenshot_20250207-094505.png` — post LinkedIn

O post argumenta que front-end não é só “alinhar div” ou “consumir back-end”. Um dev front hoje precisa lidar com:

1. **Gerenciamento de estado global** — apps com mais estados do que um país inteiro
2. **Renderização dinâmica** — SSR, SSG, ISR, Streaming; qual usar?
3. **Internacionalização (i18n)** — não é só trocar “Login” por “Iniciar Sessão”
4. **Acessibilidade (a11y)** — não opcional
5. **Integrações complexas** — GraphQL, REST, WebSockets; APIs que mudam do nada
6. **Performance e otimização** — LCP, TBT, CLS (Core Web Vitals)
7. **Segurança no client-side** — CSRF, XSS, script injection
8. **SEO e indexação** — se ninguém te acha no Google, o site existe?
9. **Gestão de design systems** — componentes reutilizáveis, acessíveis e performáticos
10. **Arquitetura de front-end** — Microfrontends, monorepos, separação de responsabilidades

Conclusão do post: front-end exige pensamento estratégico; cada decisão impacta UX, performance e acessibilidade. Desafios reais citados: botão que some no Safari; código escalável após anos de features sem refatoração.

**Enunciado:** implementar `flatten()` que achata um array profundamente aninhado.

### 2. `getMax()` — qual valor retorna?

**Fonte:** `Screenshot_20250520-113411.png` — puzzle “retorna?”

```javascript
function getMax(values) {
  let max = values[0];
  values.forEach(num => {
    if (num < max) {
      max = num;
    }
  });
  return max;
}

console.log(getMax([3, 1, 4, 2]));
```

**Pergunta:** o que `getMax([3, 1, 4, 2])` imprime?

**Resposta:** `1`. O nome sugere máximo, mas a condição `num < max` atualiza `max` com valores menores — a função retorna o **mínimo**.

**Lição:** ler a lógica, não o nome da função.

### 3. `EmailService` — code review do “sênior dinossauro”

**Fonte:** `Screenshot_20250519-123618.png` — meme de revisão de código

**Legenda:** “Os códigos que o sênior dinossauro me manda pra revisar”

```csharp
public class EmailService
{
    private readonly SmtpClient _smtp = new();

    public void SendAll(List<string> addresses)
    {
        int idx = 0;
    retry:
        if (idx >= addresses.Count) return;
        _smtp.Send(addresses[idx]);
        idx++;
        goto retry;
    }
}
```

**Desafio:** identificar problemas e reescrever de forma idiomática.

**Problemas principais:**

- Uso de `goto` em vez de `for`/`foreach`
- `_smtp.Send(addresses[idx])` — API incorreta; `SmtpClient.Send` espera `MailMessage`, não `string`
- Sem tratamento de falha por endereço (um erro interrompe o restante)
- `SmtpClient` é obsoleto no .NET moderno; preferir `MailKit` ou serviços transacionais

**Reescrita sugerida (conceitual):**

```csharp
public async Task SendAllAsync(IEnumerable<string> addresses, CancellationToken ct = default)
{
    foreach (var address in addresses)
    {
        using var message = BuildMessage(address);
        await _smtp.SendAsync(message, ct);
    }
}
```
